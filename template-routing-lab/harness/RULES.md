# Harness 规则草案

## 1. 代码边界
### 硬约束
- 本模块所有新增工作必须限制在 `template-routing-lab/` 目录内。
- `src/legacy/` 仅作为参考输入，不作为新增逻辑的长期落点。
- 新增生产代码必须放在明确职责目录中，例如：
  - `src/templates/`
  - `src/retrieval/`
  - `src/routing/`
  - `src/logging/`
  - `src/contracts/`
- 检索逻辑不得直接调用远程模型接口。
- 日志模块不得夹带路由判定逻辑。
- 测试不得依赖网络。

### 推荐流程
- schema 定义与算法实现分离。
- prompt 规范化、召回、打分、路由拆分成独立模块。
- 模板元数据与模板几何来源解耦。

## 2. 数据安全
### 硬约束
- 不得记录密钥、API token、cookie、认证头或环境变量。
- 不得无差别记录完整原始请求体，尤其是未来可能包含用户私有上下文时。
- 用户 prompt 只有在“本地开发、已记录存储位置”的前提下才能落本地日志。
- 所有持久化反馈信号必须最小化、非敏感。

### 推荐流程
- 分析日志优先使用规范化 prompt 或摘要，而不是原始全文。
- v1 的反馈优先使用枚举值，而不是自由文本。
- 日志保留策略应本地化、可清理、可关闭。

## 3. 权限控制
### 硬约束
- 本模块必须可以在本地离线运行。
- v1 的模板匹配只能使用本地模板注册表。
- 任何生成式模板提案都必须经过显式 review，才能进入正式模板库。
- 反馈日志不得自动修改模板元数据或路由阈值。

### 推荐流程
- 将生成式模板提案视为不可信输入，默认隔离。
- 候选模板进入正式模板库前必须经过人工确认。

## 4. 测试要求
### 硬约束
- 每条新增检索规则都必须至少有一个测试覆盖。
- 每次调整路由阈值都必须更新或新增测试。
- 必须为以下内容提供确定性测试：
  - prompt 规范化
  - 候选召回
  - 排序结果
  - 路由决策
  - 日志 payload 结构
- 必须有负向测试，覆盖明显不匹配的场景，例如动物模板误命中车辆需求。

### 推荐流程
- 维护一组固定 prompt fixture，覆盖常见和对抗性输入。
- 优先使用表驱动测试覆盖打分与路由。
- 每次发现误判，都要补回归测试。

## 5. Review 要求
### 硬约束
- 所有路由逻辑改动都必须附带至少一个 decision trace 样例。
- 所有 schema 改动都必须说明兼容性影响。
- 所有新增模板都必须具备完整元数据。
- 所有日志改动都必须经过隐私检查。

### 推荐 Review 问题
- 路由结果能否从元数据和 prompt 信号直接解释出来？
- 只看日志，其他工程师能否定位误路由原因？
- 此改动是否降低了确定性？
- 此改动是否让模板扩展更容易，而不是更难？
- 当前模板规模下，这个复杂度是否值得？

## 6. 质量基线
### 硬约束
- 优先保证正确和可解释，不追求“聪明但难懂”。
- 优先维护小而干净的模板库，不接受大而噪的模板堆积。
- 优先人工审核加校验，不接受生成式批量模板直接入库。

### 推荐流程
- 先优化稳定性，再扩大覆盖面。
- 只在明确有近期开销方时添加扩展点。
