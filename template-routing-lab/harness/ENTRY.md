# Harness 入口文档

## 项目目标
为体素生成系统建立第一版可工程化落地的“模板路由层”。

本模块的职责不是直接替代最终生成模型，而是在用户输入和下游体素生成之间，提供一个稳定、可解释、可扩展的中间层，负责三类决策：

- `reuse`：直接复用已有模板
- `adapt`：基于已有模板做改造
- `free_generate`：不命中模板，交给自由生成链路

## 功能目标
- 将现有预设生成器转成模板资产。
- 定义模板元数据 schema 与模板注册表。
- 实现 prompt 到模板的检索、打分与排序。
- 实现 `reuse / adapt / free_generate` 路由决策。
- 记录模板检索与路由日志，供离线分析使用。
- 为后续“人工主导 + 生成式辅助”的模板扩库预留扩展位。

## 非功能约束
- 首版必须可解释。
- 首版必须在相同输入和配置下保持确定性。
- 首版必须便于 review、调试和回归。
- 不做在线自主学习。
- 不依赖外部 AI 检索服务才能运行。

## 角色分工
### 模板库负责人
- 定义模板元数据。
- 维护模板注册表。
- 决定哪些模板可作为有效种子模板。

### 检索与路由负责人
- 实现 prompt 规范化、召回、打分、排序和路由逻辑。
- 保证路由行为可解释、可测试。

### 日志与反馈负责人
- 定义日志结构。
- 记录检索和路由过程中的证据。
- 记录轻量反馈信号，供离线分析使用。

### Reviewer
- 审查 schema 稳定性、路由正确性和测试完整性。
- 拒绝降低可解释性或缺少证据支撑的改动。

### AI 协作者
- 可以协助产出 schema、代码、测试和文档。
- 必须遵守本文档中的约束和证据标准。

## 硬约束
- v1 的模板路由必须是规则可解释的。
- 模板匹配不能依赖外部网络服务。
- 所有模板都必须有元数据，不能只有几何内容。
- 所有路由决策都必须输出机器可读的原因。
- 日志中不得记录密钥、令牌或敏感凭据。
- 反馈数据不得直接在运行时修改路由行为。

## 推荐工作流
以下是推荐流程，不是硬约束；如有偏离，必须能说明原因。

1. 以 `src/legacy/utils/voxelGenerators.ts` 中的种子生成器为起点。
2. 将每个种子生成器注册成模板，并补齐元数据。
3. 先实现 prompt 规范化与规则召回。
4. 再实现打分、排序和候选输出。
5. 再实现 `reuse / adapt / free_generate` 路由。
6. 再增加结构化日志与轻量反馈采集。
7. 在检索质量稳定前，不扩大量模板。
8. 先补测试，再扩模板。

## Evidence Standard
所有关键改动必须提供明确证据，不能只靠感觉。

可接受证据：
- 现有源码行为
- 可重复执行的测试
- 固定 prompt 的排序和路由输出样例
- 本地运行产生的结构化 decision trace
- 带前后对比的误判修复案例

不可作为唯一证据：
- “看起来更智能”
- “感觉更合理”
- 没有记录的人工试验
- 仅靠 prompt 文字说明、没有测试或 trace 支撑

## AI 应该做
- 优先复用现有种子生成器作为模板来源。
- 将规范化、召回、打分、路由拆成可组合的小模块。
- 所有决策输出都带结构化原因。
- 保持打分规则透明、易于调整。
- 为排序、阈值和路由行为编写测试。
- 让日志足以支持离线分析。
- 必要时做合理假设，并在文档或代码注释中写清楚。

## AI 不能做
- 不能引入隐藏的在线学习逻辑。
- 不能在没有测试的情况下偷偷调整路由阈值。
- 不能在日志里存储密钥、令牌或敏感环境变量。
- 不能让路由依赖随机数或不稳定状态。
- 不能绕过元数据 schema 直接写死特判逻辑。
- 不能把生成式扩展出的候选模板直接自动入库。
- 不能在 v1 用 embeddings-only 检索替代可解释规则检索。

## 完成交付的定义
只有以下内容都存在时，Harness 范围才算完整：

- 模板元数据 schema
- 模板注册表
- 检索模块
- 路由模块
- 日志 schema 与采集点
- 覆盖核心路由的测试
- `harness/` 目录下可审查的文档
